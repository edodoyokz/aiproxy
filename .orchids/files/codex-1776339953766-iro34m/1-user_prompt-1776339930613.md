Build a production-leaning MVP for a fullstack SaaS web app called "Aiproxy".

Context:
- This product is a SaaS control plane built on top of an existing proxy runtime called CLIProxyAPIPlus.
- Do NOT rebuild the proxy engine itself.
- Treat CLIProxyAPIPlus as an external runtime/service that each workspace can provision and manage through a future adapter or management API integration.
- Your job is to build the web product, control plane foundations, dashboard, auth, billing-ready plan model, and internal admin tooling.

Product goal:
- Turn a technical OAuth proxy runtime into a self-serve SaaS.
- Users should be able to sign up, create a workspace, connect an OAuth provider, generate an Aiproxy API key, see usage, and upgrade plans.
- The MVP must support tiered subscriptions starting with a freemium tier that only allows 1 OAuth proxy connection and low rate limits.

Core business rules:
- Subscription is attached to a workspace, not an individual user.
- Each user can own at least one workspace.
- Free tier:
  - 1 OAuth proxy slot only
  - 1 workspace owner
  - 1 tenant API key
  - low rate limits
  - short log retention
- Starter tier:
  - 3 OAuth proxy slots
  - 3 API keys
  - better usage dashboard
- Pro tier:
  - 10 OAuth proxy slots
  - multiple API keys
  - routing/fallback settings UI placeholders
- Team/Business can exist in the data model and pricing UI, but can be marked as “contact us” or “coming soon”

Recommended stack:
- Frontend: Next.js + TypeScript
- Backend: use a simple integrated app architecture that Orchids can generate fast, but keep the code organized so it can later be split into services
- Database: PostgreSQL
- ORM: Prisma or equivalent
- Auth: email/password with session auth
- UI: clean SaaS dashboard design, responsive, modern, not generic

Architecture constraints:
- Create the app as a control plane, not a proxy runtime
- Include clear service boundaries in the code structure:
  - auth
  - workspaces
  - subscriptions/plans
  - oauth provider connections
  - runtime provisioning adapter
  - tenant API keys
  - usage analytics
  - admin backoffice
- For anything that depends on CLIProxyAPIPlus, implement interfaces, adapters, mocks, and TODO integration points instead of fake final logic
- Keep these integration points explicit and easy to swap with real CLIProxyAPIPlus management API calls later
