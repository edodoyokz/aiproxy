# Public Launch Plan for Aiproxy

## Goal

Get Aiproxy to a credible public launch state as fast as possible without shipping known critical security or operational failures.

## Assumptions

- The near-term objective is **public launch soon**, not a long enterprise-hardening cycle.
- The fastest safe path is a **public beta / limited paid launch** with strict guardrails.
- `CLIProxyAPIPlus` remains the runtime core, and Aiproxy launches as the control plane.
- PostgreSQL is the intended production database and must become the single source of truth.

## Current Launch Verdict

The project is **not ready for public launch yet**. It is closer to an internal alpha / controlled beta.

### Critical blockers

1. Global admin authorization is unsafe.
2. Prisma migration history is not production-safe for the declared PostgreSQL target.
3. Runtime integration is not yet proven production-complete.

### High-risk gaps

- Auth hardening is minimal for public exposure.
- Billing/entitlements are operationally incomplete.
- CI is missing in practice.
- Deployment is only partially productionized.
- Observability is too weak for launch support.
- Docs and UI promise more than the code reliably delivers.

## Evidence Snapshot

- Admin routes treat normal workspace `owner` / `admin` roles as platform admin: `src/app/admin/page.tsx`, `src/app/admin/workspaces/[id]/page.tsx`, `src/app/api/admin/workspaces/route.ts`, `src/app/api/admin/workspaces/[id]/entitlement/route.ts`, `src/app/api/admin/workspaces/[id]/runtime/route.ts`
- Auth context only resolves workspace membership, not platform-level admin identity: `src/lib/authz.ts`
- Schema targets PostgreSQL but migration state appears inconsistent with that claim: `prisma/schema.prisma`, `prisma/migrations/20260416123738_init/migration.sql`
- Runtime health/readiness only checks config presence plus DB reachability, not true upstream readiness: `src/lib/runtime-health.ts`, `src/app/api/health/route.ts`, `src/app/api/ready/route.ts`
- Runtime mode defaults to mock behavior unless configured otherwise, and service layer still contains TODO-grade behaviors: `src/integrations/runtime/adapter-instance.ts`, `src/integrations/runtime/mock-adapter.ts`, `src/integrations/runtime/service.ts`
- Proxy path lacks stronger production protections and logs cost as zero: `src/app/api/proxy/chat/route.ts`, `src/lib/runtime-proxy.ts`, `src/lib/providers/proxy.ts`
- Auth throttling is in-memory and not distributed: `src/lib/auth.ts`, `src/lib/security/rate-limit.ts`
- CI workflow is absent while tests expect it to exist: `tests/ci-artifacts.test.ts`
- Deployment stack is only a baseline Docker + nginx setup: `Dockerfile`, `docker-compose.prod.yml`, `deploy/nginx.conf`, `scripts/deploy.sh`

## Launch Strategy Recommendation

Use a **two-track launch**:

### Track A — Fastest safe external launch

Launch as **public beta** with:
- free tier only, or manual invoicing only
- limited user volume
- single region
- explicit support/manual operations
- clearly labeled beta positioning

### Track B — Full commercial launch

Defer until:
- billing lifecycle is real
- runtime reliability is proven in staging
- admin/security issues are closed
- CI and deployment gates are real

## Recommended Sequence

## Phase 0 — Freeze the launch surface

**Objective:** Reduce risk by narrowing what is promised and what must work on day 1.

### Decisions

- Launch mode: `public beta` first.
- Commercial model at launch:
  - preferred: free beta
  - acceptable: manual paid activation only
- Supported provider scope:
  - preferred: one provider path first (`openai`) until runtime layer is proven
- Runtime model:
  - no public launch unless real runtime-backed flow works end to end in staging

### Deliverables

- Short launch scope statement
- List of supported flows and unsupported flows
- Public beta wording for landing page and docs

### Exit criteria

- The team agrees on exactly what is in scope for launch.
- Anything not ready is explicitly removed from launch promises.

## Phase 1 — Fix no-go security and authorization blockers

**Objective:** Eliminate the issues that make public exposure unsafe.

### Workstreams

#### 1.1 Platform admin authorization redesign

Replace workspace-role-based admin access with a real platform admin model.

Required changes:
- Introduce a platform-level admin concept separate from workspace roles.
- Gate all `/admin` pages and `/api/admin/*` routes using that platform-level authority only.
- Audit all admin data queries so they never rely on ordinary workspace membership.
- Add negative tests proving workspace owners/admins cannot access global admin surfaces.

Likely implementation options:
- simplest: env-configured allowlist of admin emails for beta
- better: `User.isPlatformAdmin` or dedicated admin membership table

Recommended beta default:
- env-configured allowlist for speed, then evolve later

#### 1.2 Auth hardening baseline

Required changes:
- Add real rate limiting for login and signup routes.
- Enforce lockout/throttling before password verification.
- Review CSRF posture for all cookie-authenticated state-changing routes.
- Add session invalidation strategy for logout and role/privilege changes.
- Ensure secure cookie settings are correct behind production proxy.

Recommended beta default:
- IP + email based rate limiting via Redis or equivalent shared store
- no public launch with in-memory-only throttling

### Exit criteria

- Normal workspace owners/admins cannot access platform admin surfaces.
- Login/signup abuse protection works across instances.
- Session behavior is documented and tested.

## Phase 2 — Repair database production readiness

**Objective:** Make database setup reproducible, safe, and aligned with PostgreSQL.

### Workstreams

#### 2.1 Reconcile Prisma schema and migrations

Required changes:
- Audit current `schema.prisma` against committed migrations.
- Create a clean migration history for PostgreSQL.
- Ensure all schema models exist in committed migrations.
- Validate both fresh bootstrap and upgrade paths.

#### 2.2 Establish one database truth

Required changes:
- Remove ambiguity around SQLite-era artifacts.
- Ensure all docs, scripts, and deploy steps refer only to PostgreSQL for production/staging.
- Make `db:migrate:deploy` the standard release path.

#### 2.3 Test migrations against real Postgres

Required changes:
- Add CI/staging check that spins up PostgreSQL and applies migrations.
- Verify Prisma client generation plus migration deployment on clean DB.

### Exit criteria

- A fresh PostgreSQL database can be migrated from zero using committed migrations.
- A staging deploy uses the same migration chain successfully.
- There is no schema-vs-migration drift.

## Phase 3 — Prove the runtime-backed core flow

**Objective:** Validate the main product promise end to end.

### Required golden path

1. User signs up.
2. Workspace is created.
3. Runtime is provisioned.
4. Provider is connected.
5. Tenant API key is issued.
6. First runtime-backed request succeeds.
7. Analytics and audit trail update correctly.

### Workstreams

#### 3.1 Runtime integration completion

Required changes:
- Audit runtime adapter behavior for real production mode.
- Remove stale mock-first assumptions from launch path.
- Add timeout, retry, and failure handling around runtime upstream calls.
- Add runtime authentication/authorization expectations between control plane and runtime.
- Define suspend/resume/recovery behavior for runtime failures.

#### 3.2 Readiness and health truthfulness

Required changes:
- `/api/health` should reflect actual component health.
- `/api/ready` should confirm deploy-readiness, not just env presence.
- Add runtime connectivity checks where safe.
- Distinguish liveness, readiness, and dependency degradation.

#### 3.3 Runtime-path analytics correctness

Required changes:
- Stop recording `cost: 0` if usage/cost is part of the product value proposition.
- Ensure request success/failure metrics are complete.
- Verify audit logging for provisioning, provider connection, key issuance, and admin actions.

### Exit criteria

- Golden path succeeds in staging without manual DB patching.
- Runtime failure modes return predictable errors.
- Health/readiness endpoints reflect reality.

## Phase 4 — Choose the monetization shape honestly

**Objective:** Avoid launching a commercial promise the system cannot yet enforce.

### Option A — Recommended for speed: Free public beta

- No self-serve billing at launch
- Manual entitlement overrides allowed internally
- Pricing page reframed as upcoming / waitlist / contact sales

### Option B — Limited paid launch

Only proceed if implemented:
- billing provider integration
- subscription lifecycle storage
- webhook processing
- entitlement sync from billing state
- grace period / failed payment handling
- support/admin override rules

### Immediate recommendation

Use **Option A** unless billing integration is already underway outside this repo.

### Exit criteria

- Product pricing, onboarding, and enforcement model all match reality.
- No user can buy something the backend cannot reliably honor.

## Phase 5 — Build a real release pipeline

**Objective:** Replace artifact/contract confidence with actual delivery confidence.

### Workstreams

#### 5.1 Restore real CI

Required changes:
- Add GitHub Actions workflow back.
- Run `npm ci`, Prisma generate, lint, typecheck, tests, build.
- Add PostgreSQL migration apply verification.
- Add at least one route-level or integration test hitting DB-backed logic.

#### 5.2 Improve test realism

Add tests for:
- admin authorization negative cases
- signup/login flows with DB
- provider connect flow with adapter mocked at boundary
- API key issuance/revocation behavior
- proxy route happy path and failure path
- health/readiness behavior under missing deps

#### 5.3 Release gating

Define release blockers:
- CI red
- migration apply failure
- staging smoke test failure
- broken golden path
- missing env vars for production

### Exit criteria

- Every deploy candidate is gated by CI.
- The app can pass a staging smoke test before production rollout.

## Phase 6 — Harden deployment and operations

**Objective:** Ensure public internet traffic can be handled safely and operably.

### Workstreams

#### 6.1 Production deployment flow

Required changes:
- Add migration deploy step into release process.
- Define exact order: build -> migrate -> start -> verify -> switch traffic.
- Add rollback instructions tied to concrete failure conditions.

#### 6.2 Reverse proxy and network hardening

Required changes:
- TLS termination strategy
- security headers validation
- request size / timeout tuning
- forwarded headers correctness
- request ID propagation
- optional gzip/compression strategy

#### 6.3 Environment and secret handling

Required changes:
- production env template completion
- secret rotation procedure
- session secret generation standard
- runtime API credential storage/rotation standard

#### 6.4 Backups and incidents

Required changes:
- tested DB backup and restore procedure
- one incident checklist for auth outage / runtime outage / DB outage
- on-call minimum response guide for launch week

### Exit criteria

- A deploy can be performed by runbook, not tribal knowledge.
- Rollback path is known and tested.
- DB restore drill is verified at least once.

## Phase 7 — Add observability that supports real incidents

**Objective:** Make launch support possible.

### Workstreams

- Replace `console.*` in critical paths with structured logging.
- Include request ID, user ID, workspace ID, runtime ID, route, status, latency, provider.
- Add error monitoring for API routes and server rendering failures.
- Add uptime checks for landing page, `/api/health`, `/api/ready`.
- Define a minimal dashboard for launch week:
  - request volume
  - auth failures
  - proxy failures
  - runtime connectivity failures
  - DB error rate

### Exit criteria

- Critical failures are diagnosable without SSH-ing into the box and guessing.
- Launch-week monitoring has named owners and thresholds.

## Phase 8 — Align product, docs, and UI with reality

**Objective:** Remove over-promising and broken paths.

### Workstreams

- Audit all dashboard and landing-page CTAs.
- Remove or implement referenced routes like missing dashboard/docs destinations.
- Rewrite README to reflect the actual launch mode.
- Update deploy and smoke-test runbooks to match exact commands.
- Add a public quickstart that matches the live onboarding path exactly.

### Exit criteria

- No visible CTA leads to a dead route.
- Docs match the launched product, not the intended future product.

## Phase 9 — Repo and release hygiene

**Objective:** Make the repository safe and credible before public attention.

### Workstreams

- Remove local artifacts from tracked history going forward.
- Ensure `.env`, build outputs, local DB artifacts, and scratch files are excluded and absent from release branches.
- Reconcile tests that expect files that are currently missing.
- Create a release checklist for final signoff.

### Exit criteria

- Release branch is clean.
- CI expectations match repo contents.

## Fastest Credible Timeline

## 72-hour emergency path

If the goal is "launch something public very soon," the minimum credible order is:

1. Fix platform admin authorization
2. Fix PostgreSQL migration chain
3. Prove staging golden path with real runtime mode
4. Add real CI workflow
5. Add distributed auth rate limiting
6. Tighten deploy/runbook flow
7. Launch as free public beta with limited traffic

## 1-2 week practical path

### Week 1
- Phase 0 scope freeze
- Phase 1 security/admin fixes
- Phase 2 DB migration repair
- Phase 3 runtime golden path in staging
- Phase 5 CI restoration

### Week 2
- Phase 6 deploy hardening
- Phase 7 observability
- Phase 8 docs/UI cleanup
- Phase 9 repo hygiene
- public beta launch

## Recommended Public Beta Definition of Done

A public beta can launch when all of the following are true:

- Platform admin is separated from workspace roles.
- PostgreSQL migrations are reproducible from scratch.
- Signup -> provider connect -> tenant key -> first successful request works in staging.
- Runtime health/readiness endpoints are truthful.
- Auth routes have real distributed abuse protection.
- CI runs on every push/PR.
- Production deploy runbook is tested once end to end.
- Structured logs and basic error monitoring are in place.
- Docs and UI reflect actual supported capabilities.

## Items That Can Wait Until After Beta

These should not block a carefully limited beta if the above is done:
- self-serve billing
- multi-provider breadth beyond one stable provider
- advanced analytics polish
- enterprise-grade session management
- deep autoscaling sophistication
- multi-region runtime placement

## Execution Order Inside the Repo

Suggested implementation order for lowest rework:

1. Authorization model and admin route guards
2. Prisma migration repair and DB verification
3. Runtime integration stabilization and golden-path test
4. Auth rate limiting and session hardening
5. CI workflow restoration
6. Deployment and readiness hardening
7. Logging/monitoring wiring
8. Docs, UI CTA cleanup, pricing/launch messaging
9. Repo cleanup and final release checklist

## Final Recommendation

Do **not** launch this as a fully commercial public SaaS yet.

Launch it as a **public beta** only after closing the admin auth issue, migration drift, and runtime proof gaps. That path is realistic, fast, and honest. Once those are fixed, the project can plausibly ship publicly with manual operations and a narrow scope, then iterate toward full paid launch from a much safer base.
